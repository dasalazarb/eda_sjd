## Project overview

This repository contains a sequentially numbered pipeline for longitudinal clinical data analysis.
The data originates from two NIH clinical protocols (11-D-0172 and 15-D-0051) involving a
Sjögren's Disease natural history cohort. The pipeline ingests raw CTDB exports, harmonizes a
dual-protocol codebook, builds longitudinal visit structures, audits data quality, and produces
analysis-ready datasets for downstream modeling (e.g., ESSDAI progression, cohort viability,
missingness analysis).

**Primary language:** Python 3.10+  
**Execution environment:** NIH Biowulf HPC (SLURM), with local VSCode for development  
**Core libraries:** pandas, polars, scikit-learn, XGBoost, openpyxl, matplotlib, seaborn

## Script numbering and output traceability

**This is a hard architectural rule. Do not deviate from it.**

- Every script in `src/` is numbered with a two-digit prefix (e.g., `07_build_cohorts.py`).
  New scripts must follow the next available number in sequence.
- Every output file produced by a script must be named so it can be traced back to its origin.
  Use the script number as a prefix or include it in a subdirectory name.

  Examples:
  - Script `13_longitudinal_plausibility_audit.py` → outputs go to
    `reports/longitudinal_plausibility/` or are named `13_*.csv`
  - Script `08_visit_patterns.py` → outputs go to `reports/visit_patterns/`

- Never produce outputs that are ambiguously named or lack traceability to their generating script.
- When adding a new script, update `00_super_main.py` only if the new step belongs in the main
  pipeline execution order.
- The `common.py` module contains shared utilities (paths, logging helpers, I/O wrappers). Check it
  before implementing anything that might already exist there.

## Execution environment (Biowulf HPC)

This pipeline runs on NIH Biowulf under SLURM. Key environment facts:

- Python is loaded via environment modules: `module load python/3.10`
- All dependencies are installed in a project virtualenv. Do not assume system-wide packages.
- Do not hardcode absolute paths. Use `pathlib.Path` and derive paths from a project root constant
  defined in `common.py`.
- SLURM job submissions go through `run_00_super_main.sh`. Do not add ad-hoc `sbatch` calls inside
  Python scripts.
- Heavy computation (e.g., model training, large-scale joins) must be wrapped in functions that can
  be called from a SLURM batch context without interactive terminal input.
- Do not use `input()` or `getpass()` in any pipeline script.

### Running the pipeline

```bash
# Load environment
module load python/3.10
source /path/to/venv/bin/activate

# Run full pipeline
python src/00_super_main.py

# Run individual step
python src/07_build_cohorts.py

# Submit to SLURM
sbatch src/run_00_super_main.sh
```

## Data and clinical context

- The cohort spans two NIH protocols: **11-D-0172** and **15-D-0051** (Sjögren's Disease natural
  history study).
- Raw data lives exclusively in `data_raw/`. It must never be modified, overwritten, or deleted.
- Intermediate and analytic datasets are derived outputs. They can be regenerated from `data_raw/`.
- Key clinical constructs to be aware of:
  - **ESSDAI** (EULAR Sjögren's Syndrome Disease Activity Index) — multi-domain score, exists in
    two protocol versions that require harmonization (see `09b_merge_essdai_versions.py`).
  - **ESSPRI** — patient-reported symptom index.
  - **Intervals** — visits are structured into named longitudinal intervals (Baseline, Y1, Y2, …).
    Temporal ordering and plausibility are audited in scripts 12–14.
  - **BTRIS** — NIH clinical data system; matched visits are in `data_analytic/BTRIS/`.
  - **Codebook** — dual-protocol harmonized codebook in `data_raw/codebook_final_harmonized_*.xlsx`.
    All variable names and value recodes must reference this codebook.

## Hard constraints

These constraints are non-negotiable. Violating them is grounds to abort the task.

1. **Never modify files under `data_raw/`** — this is the source of truth. Read only.
2. **Never commit automatically.** Prepare changes and present a diff or summary for human review.
   Do not run `git commit`, `git push`, or `git merge` without explicit instruction.
3. **Never overwrite a script's output file with unverified data.** If a script produces
   `data_analytic/cohort_baseline.parquet`, do not overwrite it unless the generating script has
   been run and validated.
4. **Never introduce new Python dependencies** without listing them and explaining why no existing
   library in the project covers the need.
5. **Never rename columns, output filenames, or public function signatures** without flagging the
   change explicitly and confirming it will not break downstream scripts.
6. **Before large or destructive changes, state the plan and wait for confirmation.**
   Large = touching more than one numbered script, renaming files, restructuring directories, or
   changing shared logic in `common.py`.

## Code conventions

### General style

- Follow the existing code style in each file before introducing new patterns.
- Use `black`-compatible formatting (88-char line limit). If `ruff` or `black` config is added
  to the repo, run it before submitting changes.
- Organize imports: stdlib → third-party → local, with a blank line between groups. Remove unused
  imports.
- Prefer explicit over implicit. Avoid "magic" one-liners that sacrifice readability.
- Keep functions small and single-purpose. If a function exceeds ~40 lines, consider splitting it.

### Type hints and docstrings

- Add type hints to all new or modified function signatures.
- Write NumPy-style docstrings for all public functions, classes, and modules.

```python
def collapse_visits(
    df: pd.DataFrame,
    interval_col: str = "interval_name",
    subject_col: str = "subject_id",
) -> pd.DataFrame:
    """
    Collapse multiple rows per subject-interval into one row.

    Parameters
    ----------
    df : pd.DataFrame
        Long-format visits dataframe with one row per observation.
    interval_col : str
        Column identifying the visit interval.
    subject_col : str
        Column identifying the patient.

    Returns
    -------
    pd.DataFrame
        Wide-format dataframe with one row per subject-interval.
    """
```

### Naming conventions

- Files: `NN_descriptive_snake_case.py` (numbered scripts) or `descriptive_snake_case.py` (utils).
- Functions and variables: `snake_case`.
- Classes: `PascalCase`.
- Constants: `UPPER_SNAKE_CASE`, defined at module top or in `common.py`.
- Output files: prefix with the script number or store in a subdirectory named after the script's
  domain (e.g., `reports/visit_patterns/`, `reports/missingness/`).

### Path handling

```python
# Always use pathlib. Never hardcode absolute paths.
from pathlib import Path
from src.common import PROJECT_ROOT   # or define locally if needed

DATA_RAW = PROJECT_ROOT / "data_raw"
DATA_ANALYTIC = PROJECT_ROOT / "data_analytic"
REPORTS = PROJECT_ROOT / "reports"
```

### Logging

- Use the shared logger from `common.py` (or Python's `logging` module if none exists yet).
- Do not use bare `print()` in pipeline scripts. Use `logger.info()`, `logger.warning()`,
  `logger.error()`.
- Log the start and end of each major step, including input shapes and output paths.

### I/O

- Prefer `.parquet` for intermediate and analytic datasets (faster, typed). Keep `.csv` copies
  only when human readability is needed for audit trails.
- Always write outputs to the correct directory tier:
  - `data_intermediate/` for in-progress transformations
  - `data_analytic/` for final analysis-ready tables
  - `reports/` for figures, audit CSVs, and Excel summaries

## Autonomous task guidelines

The agent may perform the following tasks without additional prompting:

- **Read and understand** the full pipeline structure by inspecting `src/`, `common.py`, and
  `data_analytic/` before making changes.
- **Refactor** functions or scripts for readability, modularity, and DRY compliance, without
  altering outputs.
- **Debug** pipeline failures using logs in `logs/` and tracebacks from SLURM `.err` files.
- **Add or update unit tests** for functions in `common.py` or any script being modified.
- **Add or improve docstrings** and inline documentation.
- **Create new numbered scripts** following the naming convention, with all outputs traceable to
  the new script number.
- **Improve logging and error messages** for clarity and debuggability.
- **Propose** technical debt items, redundant code, or missing validations with a clear explanation.

The agent must **not** autonomously:

- Run the full pipeline or any script that writes to `data_analytic/` or `data_raw/`.
- Delete or rename any existing output file without a confirmed plan.
- Modify `00_super_main.py` (the orchestrator) without explaining the impact on execution order.
- Introduce new analysis logic (new statistical models, new cohort definitions) without confirming
  intent with the user.
